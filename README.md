# CS230-Portfolio
Software Design Document for the Gaming Room

Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?  
The Gaming Room wanted to expand Draw It or Lose It into a web‑based, multi‑platform game. They needed support for multiple teams, no duplicate game sessions, and a system that could scale across different operating systems.

What did you do particularly well in developing this documentation?  
I organized the design clearly and explained the technical decisions in a way that made sense for both developers and the client.

What about the process of working through a design document did you find helpful when developing the code?  
Having the design laid out first gave me a solid plan. It helped me understand the structure of the system before writing any code and kept me focused on the actual requirements.

If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?  
I would revise the architecture section and add clearer diagrams or examples to show how the components interact.

How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?  
I focused on what the client wanted the game to do—support multiple players, avoid duplicate sessions, and run smoothly across platforms. Designing around user needs ensures the final product actually solves their problem.

How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?  
I broke the project into smaller parts like requirements, constraints, and architecture. In the future, I’d use the same structured approach along with diagrams and use cases to stay organized.
